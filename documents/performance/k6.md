# k6 — Performance Testing Guideline

> The **default** perf-testing tool for this repo. JavaScript-authored, threshold-native, CI-friendly. Used by the [`performance-testing` skill](../../.agents/skills/performance-testing/SKILL.md) (Lane 1) and the [`performance-analyzer` skill](../../.agents/skills/performance-analyzer/SKILL.md).
>
> Read [`README.md`](./README.md) first for the shared discipline (SLOs, dashboard contract, decision matrix). This doc is the **k6-specific implementation** of that contract.

## TL;DR

- Author scenarios in JavaScript under `tests/perf/k6/scenarios/`.
- One file per scenario; one SLO per file; one owner per file.
- `k6 run --summary-export` writes JSON; `scripts/perf-summary.ts` collapses it into a dashboard row.
- CI gates on threshold breach (k6 exits non-zero); cron job refreshes baselines.

## Install

| Surface | Command | Pinned version |
|---|---|---|
| macOS local | `brew install k6` | `>= 0.51` |
| Linux local | `sudo gpg -k && sudo gpg --no-default-keyring --keyring /usr/share/keyrings/k6-archive-keyring.gpg --keyserver hkp://keyserver.ubuntu.com:80 --recv-keys C5AD17C747E3415A3642D57D77C6C491D6AC1D69 && echo "deb [signed-by=/usr/share/keyrings/k6-archive-keyring.gpg] https://dl.k6.io/deb stable main" \| sudo tee /etc/apt/sources.list.d/k6.list && sudo apt update && sudo apt install k6` | `>= 0.51` |
| Docker | `docker run --rm -i grafana/k6:0.51.0 run - <scenario.js` | `0.51.0` (pin per release) |
| CI (GitHub Actions) | `grafana/setup-k6-action@v1` | (uses latest unless `version:` set) |

Pin the version in CI explicitly — k6's threshold semantics have shifted between minors.

## Hello world — `cart-add-item.js`

```javascript
// OWNER: @khanhdo
// SCENARIO: cart-add-item
// SLO: P95 < 500ms, error_rate < 0.5%, sustained 50 VUs for 5 min

import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate } from 'k6/metrics';
import { ENV } from '../lib/env.js';
import { login } from '../lib/auth.js';
import { writeSummary } from '../lib/summary.js';

const cartErrors = new Rate('cart_add_errors');

export const options = {
  scenarios: {
    cart_add_item: {
      executor: 'ramping-vus',
      startVUs: 0,
      stages: [
        { duration: '30s', target: ENV.VUS },   // ramp
        { duration: ENV.DURATION, target: ENV.VUS }, // steady
        { duration: '30s', target: 0 },              // ramp-down
      ],
      gracefulRampDown: '10s',
    },
  },
  thresholds: {
    // SLO 1 — latency
    'http_req_duration{name:cart_add}': ['p(95)<500'],
    // SLO 2 — error rate
    'cart_add_errors': ['rate<0.005'],
    // SLO 3 — at least N successful adds (sanity floor)
    'http_reqs{name:cart_add,expected_response:true}': ['count>=1000'],
  },
  tags: { service: 'cart', scenario: 'cart-add-item', env: ENV.ENV_NAME },
};

let token;
export function setup() {
  return { token: login(ENV.BASE_URL, ENV.USERNAME, ENV.PASSWORD) };
}

export default function (data) {
  const res = http.post(
    `${ENV.BASE_URL}/api/cart/items`,
    JSON.stringify({ productId: 'SKU-42', qty: 1 }),
    {
      headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${data.token}` },
      tags: { name: 'cart_add' },
    },
  );
  const ok = check(res, {
    'status is 200': (r) => r.status === 200,
    'has cart_id': (r) => !!r.json('cart_id'),
  });
  cartErrors.add(!ok);
  sleep(1);
}

export function handleSummary(data) {
  return writeSummary(data, { scenario: 'cart-add-item', service: 'cart', tool: 'k6' });
}
```

Three things this template enforces:

1. **Header is mandatory** — owner, scenario name, SLO. The CI lint job (planned) refuses files without all three.
2. **Thresholds drive the exit code** — k6 exits 99 (configurable) when any threshold fails; CI sees that as build-failure.
3. **`handleSummary` writes to the dashboard contract** — never let k6's default output be the artifact; always pipe through `lib/summary.js`.

## `lib/env.js` — env var contract

```javascript
const REQUIRED = ['BASE_URL', 'VUS', 'DURATION', 'ENV_NAME'];

export const ENV = {
  BASE_URL: __ENV.BASE_URL || 'https://qa.ecommerce-playground.lambdatest.io',
  VUS: parseInt(__ENV.VUS || '50', 10),
  DURATION: __ENV.DURATION || '5m',
  RAMP: __ENV.RAMP || '30s',
  ENV_NAME: __ENV.ENV_NAME || 'qa',
  USERNAME: __ENV.PERF_USER || 'perf-bot@example.com',
  PASSWORD: __ENV.PERF_PASS || 'changeme',
};

const missing = REQUIRED.filter((k) => !ENV[k]);
if (missing.length) throw new Error(`k6 missing env vars: ${missing.join(', ')}`);
```

**Never hardcode `BASE_URL`.** The same script runs against `qa`, `staging`, `prod-shadow` via env vars only.

## `lib/summary.js` — dashboard adapter

```javascript
import { textSummary } from 'https://jslib.k6.io/k6-summary/0.0.2/index.js';
import { ENV } from './env.js';

export function writeSummary(data, meta) {
  const trend = data.metrics['http_req_duration'];
  const reqs = data.metrics['http_reqs'];
  const fail = data.metrics['http_req_failed'];

  const row = {
    scenario: meta.scenario,
    tool: meta.tool,
    service: meta.service,
    env: ENV.ENV_NAME,
    p50_ms: round(trend.values['p(50)']),
    p95_ms: round(trend.values['p(95)']),
    p99_ms: round(trend.values['p(99)']),
    error_rate: round(fail.values.rate, 4),
    throughput_rps: round(reqs.values.rate),
    duration_sec: data.state.testRunDurationMs / 1000,
    vus_peak: data.metrics['vus_max'].values.max,
    slo_passed: Object.values(data.metrics).every((m) => !m.thresholds || Object.values(m.thresholds).every((t) => !t.fails)),
    ts_iso: new Date().toISOString(),
    owner: '@khanhdo',
    baseline_ref: `reports/perf/baselines/${ENV.ENV_NAME}/${meta.scenario}.json`,
  };

  return {
    'stdout': textSummary(data, { indent: ' ', enableColors: true }),
    [`reports/perf-summary-${meta.scenario}.json`]: JSON.stringify(row, null, 2),
  };
}

const round = (v, p = 2) => Math.round(v * 10 ** p) / 10 ** p;
```

`scripts/perf-summary.ts` (referenced in [`README.md`](./README.md)) merges every `reports/perf-summary-*.json` into a single `reports/perf-summary.json` for the dashboard.

## Local run

```bash
# default — qa env, 50 VUs, 5 min
k6 run tests/perf/k6/scenarios/cart-add-item.js

# different env / load
BASE_URL=https://staging.example.com VUS=100 DURATION=10m ENV_NAME=staging \
  k6 run tests/perf/k6/scenarios/cart-add-item.js

# inline summary view
k6 run --summary-trend-stats="avg,min,med,max,p(95),p(99)" tests/perf/k6/scenarios/cart-add-item.js
```

Exit codes:

| Code | Meaning |
|---|---|
| `0` | All thresholds passed |
| `99` | One or more thresholds failed (CI must treat as failure) |
| `108` | Script error (parse / runtime exception) |

## CI — GitHub Actions

Add a job to `.github/workflows/playwright.yml` (or a sibling `perf.yml`). Mirrors the same three-env matrix shape used by the main pipeline (see [`documents/ci/github-actions.md`](../ci/github-actions.md)).

```yaml
perf-k6:
  if: github.event_name == 'pull_request' && contains(github.event.pull_request.changed_files, 'pages/ui/cart-page.ts')
  runs-on: ubuntu-latest
  needs: install
  strategy:
    fail-fast: false
    matrix:
      scenario: [cart-add-item, checkout-submit, search-query]
      env: [qa]   # extend to staging on cron, never to prod
  steps:
    - uses: actions/checkout@v4
    - uses: grafana/setup-k6-action@v1
      with:
        k6-version: '0.51.0'
    - name: Run k6 — ${{ matrix.scenario }} on ${{ matrix.env }}
      env:
        BASE_URL: ${{ secrets[format('BASE_URL_{0}', matrix.env)] }}
        ENV_NAME: ${{ matrix.env }}
        VUS: '50'
        DURATION: '5m'
        PERF_USER: ${{ secrets.PERF_USER }}
        PERF_PASS: ${{ secrets.PERF_PASS }}
      run: k6 run tests/perf/k6/scenarios/${{ matrix.scenario }}.js
    - name: Upload summary
      if: always()
      uses: actions/upload-artifact@v4
      with:
        name: perf-k6-${{ matrix.scenario }}-${{ matrix.env }}
        path: reports/perf-summary-${{ matrix.scenario }}.json
        retention-days: 30
```

Two CI shapes:

| Shape | Trigger | Action on threshold fail |
|---|---|---|
| **PR gate** | `pull_request` touching a hot file | Block merge |
| **Nightly baseline** | `schedule:` cron on `main` | Refresh baselines via PR; page on regression > 20 % |

## Updating baselines

Baselines live in `tests/perf/k6/baselines/<env>/<scenario>.json`. Update flow:

```bash
# 1. Run on a quiet env (or nightly)
ENV_NAME=staging k6 run tests/perf/k6/scenarios/cart-add-item.js
# 2. Promote the summary to the baseline
mv reports/perf-summary-cart-add-item.json tests/perf/k6/baselines/staging/cart-add-item.json
# 3. PR with title "perf(baseline): refresh cart-add-item @ staging — <reason>"
```

**Never update a baseline silently.** Every refresh is a PR with a one-line justification (deploy that improved the path, infra upgrade, intentional algorithmic change). Drift goes through review like code.

## Output → Grafana / Prometheus / InfluxDB

Optional but recommended for any team running > 5 perf scenarios.

```bash
# Prometheus remote-write (k6 v0.42+)
K6_PROMETHEUS_RW_SERVER_URL=http://prom.local/api/v1/write \
  k6 run --out experimental-prometheus-rw tests/perf/k6/scenarios/cart-add-item.js

# InfluxDB v2
K6_INFLUXDB_TOKEN=$INFLUXDB_TOKEN \
  k6 run --out xk6-influxdb=http://influx.local:8086/k6 tests/perf/k6/scenarios/cart-add-item.js
```

The dashboard contract (`reports/perf-summary.json`) and the metrics output are independent — emit both. The summary is the **build-time gate**; the time-series is the **drift detector** between runs.

## Anti-patterns specific to k6

| Anti-pattern | Why it's bad | Do this instead |
|---|---|---|
| `sleep(0)` between requests | Generates unrealistic per-VU throughput; saturates the test rig before the SUT | `sleep(1)` minimum; tune to traffic shape |
| Defining thresholds inline as `'p(95)<500'` strings without a metric tag | Threshold matches *all* requests, not the targeted one | Always tag the request: `tags: { name: 'cart_add' }` and threshold `http_req_duration{name:cart_add}` |
| One mega-scenario hitting 8 endpoints | Threshold semantics merge across endpoints; you can't tell which is slow | One scenario per endpoint or business action |
| `check()` without contributing to a metric | Failed checks don't fail the build by default | Pair every `check` with a `Rate` metric and threshold it |
| `discardResponseBodies: true` for a body-validating test | Saves memory, breaks the assertions | Only set when you literally don't read the body |
| Using k6 cloud secrets in a fork PR | k6 cloud token leaks to attacker | Run cloud only on `main` and protected branches; PR runs are local-engine only |
